142 /*****************************************************************************************************************************************************
143  * NOTES:
144  *
145  * 1. The single-sided two-way ranging scheme implemented here has to be considered carefully as the accuracy of the distance measured is highly
146  * sensitive to the clock offset error between the devices and the length of the response delay between frames. To achieve the best possible
147  * accuracy, this response delay must be kept as low as possible. In order to do so, 6.8 Mbps data rate is used in this example and the response
148  * delay between frames is defined as low as possible. The user is referred to User Manual for more details about the single-sided two-way ranging
149  * process.
150  * 2. The sum of the values is the TX to RX antenna delay, experimentally determined by a calibration process. Here we use a hard coded typical value
151  * but, in a real application, each device should have its own antenna delay properly calibrated to get the best possible precision when performing
152  * range measurements.
153  * 3. The frames used here are Decawave specific ranging frames, complying with the IEEE 802.15.4 standard data frame encoding. The frames are the
154  * following:
155  * - a poll message sent by the initiator to trigger the ranging exchange.
156  * - a response message sent by the responder to complete the exchange and provide all information needed by the initiator to compute the
157  * time-of-flight (distance) estimate.
158  * The first 10 bytes of those frame are common and are composed of the following fields:
159  * - byte 0/1: frame control (0x8841 to indicate a data frame using 16-bit addressing).
160  * - byte 2: sequence number, incremented for each new frame.
161  * - byte 3/4: PAN ID (0xDECA).
162  * - byte 5/6: destination address, see NOTE 4 below.
163  * - byte 7/8: source address, see NOTE 4 below.
164  * - byte 9: function code (specific values to indicate which message it is in the ranging process).
165  * The remaining bytes are specific to each message as follows:
166  * Poll message:
167  * - no more data
168  * Response message:
169  * - byte 10 -> 13: poll message reception timestamp.
170  * - byte 14 -> 17: response message transmission timestamp.
171  * All messages end with a 2-byte checksum automatically set by DW1000.
172  * 4. Source and destination addresses are hard coded constants in this example to keep it simple but for a real product every device should have a
173  * unique ID. Here, 16-bit addressing is used to keep the messages as short as possible but, in an actual application, this should be done only
174  * after an exchange of specific messages used to define those short addresses for each device participating to the ranging exchange.
175  * 5. This timeout is for complete reception of a frame, i.e. timeout duration must take into account the length of the expected frame. Here the value
176  * is arbitrary but chosen large enough to make sure that there is enough time to receive the complete response frame sent by the responder at the
177  * 6.8M data rate used (around 200 Âµs).
178  * 6. In a real application, for optimum performance within regulatory limits, it may be necessary to set TX pulse bandwidth and TX power, (using
179  * the dwt_configuretxrf API call) to per device calibrated values saved in the target system or the DW1000 OTP memory.
180  * 7. dwt_writetxdata() takes the full size of the message as a parameter but only copies (size - 2) bytes as the check-sum at the end of the frame is
181  * automatically appended by the DW1000. This means that our variable could be two bytes shorter without losing any data (but the sizeof would not
182  * work anymore then as we would still have to indicate the full length of the frame to dwt_writetxdata()).
183  * 8. We use polled mode of operation here to keep the example as simple as possible but all status events can be used to generate interrupts. Please
184  * refer to DW1000 User Manual for more details on "interrupts". It is also to be noted that STATUS register is 5 bytes long but, as the event we
185  * use are all in the first bytes of the register, we can use the simple dwt_read32bitreg() API call to access it instead of reading the whole 5
186  * bytes.
187  * 9. The high order byte of each 40-bit time-stamps is discarded here. This is acceptable as, on each device, those time-stamps are not separated by
188  * more than 2**32 device time units (which is around 67 ms) which means that the calculation of the round-trip delays can be handled by a 32-bit
189  * subtraction.
190  * 10. The user is referred to DecaRanging ARM application (distributed with EVK1000 product) for additional practical example of usage, and to the
191  * DW1000 API Guide for more details on the DW1000 driver functions.
